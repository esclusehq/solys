<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Agent → Gateway WSS | Agent authenticates with `Authorization: Bearer {relay_token}`. Gateway validates token by calling backend's HMAC-signed POST /internal/relay/authorize. |
| Agent → Backend WS | Agent sends TunnelConnect with `server_id` and `agent_public_ip`. Backend verifies ownership in Plan 03. |
| Local TCP forwarder | `relay_session::run_relay_session` opens 127.0.0.1:25565 (or configured MC address). No external network access from the yamux stream target. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-68-02 | Spoofing | Authorization Bearer token | mitigate | Token is a UUIDv4 with 122 bits of entropy. Gateway verifies it via HMAC-signed backend callback on every reconnect. |
| T-68-03 | Tampering | WSS messages | mitigate | TLS 1.3 end-to-end (D-03). Yamux control stream is inside the TLS tunnel. |
| T-68-04 | Repudiation | TunnelConnect/Disconnect | mitigate | Both events logged via `log_relay_tunnel_event` to the audit log with node_id, server_id, reason, timestamp. |
| T-68-05 | Information Disclosure | agent_public_ip | mitigate | Sent only inside the encrypted WSS to the gateway; never broadcast or returned by any REST API. |
| T-68-06 | Denial of Service | Rapid reconnect loop | mitigate | Exponential backoff with 30s cap and 20% jitter prevents thundering herd. Gateway also enforces 1 active tunnel per server_id (D-21). |
| T-68-07 | Elevation of Privilege | relay.remove_cname_record task | mitigate | Task is dispatched only from backend's `ModeOverrideChange` flow, which is itself auth-gated by the dashboard JWT. |
| T-68-08 | Information Disclosure | local TCP forward target | mitigate | The forwarder reads `state.config.relay.local_mc_addr` from agent config; user cannot override it via any inbound WS message. |

## ASVS L1 Mappings (Phase 68 agent tier only — full coverage lives in plans 03, 04)

- **V2.1 Authentication:** Bearer token is per-node, stored in `nodes.relay_token` (database-controlled), never user-supplied.
- **V3.4 Session Management:** WebSocket sessions have a 10s heartbeat with reconnect on miss; no session fixation possible (each reconnect gets a fresh yamux session).
- **V6.2 Cryptographic Practices:** TLS 1.3 only (the tokio-tungstenite client refuses TLS 1.2 by default in modern rustls configurations).
- **V7.1 Error Handling:** Reconnect backoff prevents log spam on persistent failures (3+ misses trigger a `tracing::error!`).
</threat_model>
