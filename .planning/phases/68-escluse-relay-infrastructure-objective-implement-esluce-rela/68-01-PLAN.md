<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Agent → Backend WebSocket | Agent sends TunnelConnect/Disconnect/Heartbeat with `server_id` and `agent_public_ip`. Backend trusts the `server_id` claim only if it can correlate it to a server row owned by the same `node_id` that is sending it. |
| Plan 02 agent | The agent will send `TunnelConnect` with `agent_public_ip`. The backend must verify that `server_id` belongs to the same node that owns the WebSocket connection, otherwise a malicious node could claim ownership of another tenant's server. |
| Plan 04 gateway | The gateway will lookup `server_id` from the player source IP. Pitfall 9: MC Java TCP has no SNI, so the gateway must trust the agent's `agent_public_ip` claim as the sole authority. This is acceptable because (a) the agent's relay_token is HMAC-verified at gateway startup, and (b) the tunnel yamux control stream is TLS 1.3 encrypted. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-68-01 | Spoofing | TunnelConnect.agent_public_ip | mitigate | Plan 03 verifies the sender's `node_id` (from WebSocket auth) matches the server's owner before accepting the TunnelConnect and updating `relay_status`. |
| T-68-02 | Tampering | NodeMessage enum | mitigate | All WebSocket messages are TLS 1.3 encrypted (existing). New variants carry no PII; the only identifier is `server_id` (already a UUID). |
| T-68-03 | Repudiation | TunnelDisconnect | mitigate | Plan 03 logs every TunnelDisconnect to `audit_log` with `node_id`, `server_id`, `reason`, and timestamp. |
| T-68-04 | Information Disclosure | relay_token column | mitigate | `nodes.relay_token` is marked `UNIQUE` and never returned by any API endpoint. The only consumer is the gateway (via `find_by_relay_token`) which authenticates the caller via HMAC. |
| T-68-05 | Denial of Service | Duplicate relay_token issuance | mitigate | The unique partial index `uq_nodes_relay_token` prevents two nodes from sharing a token. |
| T-68-06 | Elevation of Privilege | connectivity_mode_override column | mitigate | The CHECK constraint restricts values to `('relay', 'direct', 'auto')`; any other value (including NULL which means "no override") is rejected at the database level. |

## ASVS L1 Mappings (Phase 68 only — full coverage lives in plans 02-05)

- **V1.4 Access Control:** The relay_token is only ever written by the backend on first node registration; it is never user-settable.
- **V4.1 Input Validation:** All enum fields (`mode`, `relay_status`, `reason`) are CHECK-constrained at the database and validated at the entity layer.
- **V7.1 Logs:** `last_tunnel_disconnect_reason` and `relay_status` transitions are observable via the existing audit log (Plan 03 wires the actual writes).
