**Permission model design (per D-09/D-11/D-12):**
- `permission_overrides` table maps (guild_id, command_name, role_id) -> allowed boolean
- A command with zero override rows = admin only (restrictive default per D-11)
- A command with any allowed=true row = those roles can use it
- Admin check: interaction.member.permissions.has(PermissionFlagsBits.Administrator) OR guild owner
- Permission check runs in command-handler.ts before command.execute()

    Implement the permission check system per D-09/D-11/D-12. This gates every command execution.

    **Part A: Permission utility (`bot/src/commands/_shared/permissions.ts`)**

    ```typescript
    import { ChatInputCommandInteraction, PermissionFlagsBits } from 'discord.js';
    import { apiClient } from '../../api/client';

    // The single permission entry point. Called by command-handler before execute().
    export async function hasPermission(
      interaction: ChatInputCommandInteraction,
      commandName: string
    ): Promise<boolean> {
      // 1. Guild owner always has access
      if (interaction.guild?.ownerId === interaction.user.id) return true;

      // 2. Server-wide Administrator permission always has access
      const member = await interaction.guild?.members.fetch(interaction.user.id);
      if (member?.permissions.has(PermissionFlagsBits.Administrator)) return true;

      // 3. Check custom permission_overrides table via API
      const roleIds = member?.roles.cache.map(r => r.id) ?? [];
      const result = await apiClient.checkPermission(
        interaction.guildId!,
        commandName,
        roleIds
      );

      // 4. Restrictive default per D-11: if no rows exist, deny (admin-only)
      return result;
    }

    // Cached permission check (with Redis in later phases — in-memory for now)
    // Cache key: `${guildId}:${commandName}:${userId}` — TTL 30s
    // For Phase 1, use a simple Map<string, {result: boolean, expires: number}>
    ```

    **Key behavior:**
    - Server owner bypasses all permission checks
    - Administrator permission bypasses all checks
    - If `permission_overrides` has no rows for (guildId, commandName) → deny (admin-only)
    - If any of the user's roles has `allowed=true` → allow
    - If none of the user's roles has `allowed=true` → deny

    **Part B: Integrate into command handler (`bot/src/handlers/command-handler.ts`)**

    Modify `handleCommand()` to add a permission gate BEFORE calling command.execute():

    ```typescript
    // In handleCommand(), before execute():
    const hasPerm = await hasPermission(interaction, commandName);
    if (!hasPerm) {
      await interaction.reply({
        content: 'You do not have permission to use this command.',
        ephemeral: true,
      });
      return;
    }
    ```

    **Permission metadata on commands:** Each command can optionally declare `requiredPermissions: string[]` for role names (future use). For now, all commands are gated by the same permission_overrides table.

    **Denied response:** Ephemeral "You do not have permission to use this command." with error embed style.

    **Caching:** Add a simple in-memory cache (Map with 30s TTL) for permission results to avoid API calls on rapid re-execution. This is a non-persistent cache — bot restart clears it.

    **Do NOT:**
    - Do NOT implement Discord's native permission system (per D-12) — always use custom DB
    - Do NOT skip the permission check for any command (all commands must gate)
    - Do NOT make the cache persistent (Redis caching comes in later phases)
    - Do NOT forget to handle the case where the guild is not cached (fetch if needed)
  </action>
