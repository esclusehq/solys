<objective>
Fix 404 error when loading official templates for edit/delete.

Root cause: `get_template_by_id` query in SqlxTemplateRepository is missing the `version` column from its SELECT clause. The `Template` struct expects this field, causing a SQLx decode error at runtime. The handler silently maps ALL errors to 404, so the decode error appears as "Failed to load template" toast.

Fix: Add migration for `version` column, update the SELECT query, and differentiate error types in the handler.
</objective>

<plan_notes>
The `version` field exists on the Rust `Template` struct but was never added to the DB schema or included in the SELECT query. This causes a `ColumnNotFound` error at runtime when `get_template_by_id` runs. The `usage_count` field has `#[sqlx(default)]` so it's handled, but `version` is `Option<String>` without `#[sqlx(default)]` — sqlx raises a decode error.

All three handlers that call `get_template_by_id` internally (get, update, delete) are affected. The update and delete handlers have slightly better error mapping but still rely on the repository SQL working correctly.

The `create_template` handler (line 73) is NOT affected — it uses `INSERT ... RETURNING` which doesn't have the column mismatch issue because `version` is `None` and the RETURNING clause doesn't include it.
</plan_notes>
