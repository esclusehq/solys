# Phase 85: Perbaiki Edit dan Delete Template — Research

**Researched:** 2026-06-16
**Domain:** Backend API, Rust/axum, SQLx, Template CRUD
**Confidence:** HIGH

## Summary

Bug: Admin/owner users get 404 "Failed to load template" when trying to edit or delete official (built-in) templates via the template library UI. The toast appears because `GET /api/v1/templates/:id` returns 404.

Root cause identified: The `get_template_by_id` SQL query in `SqlxTemplateRepository` is missing `version` and `usage_count` columns from its SELECT clause. These columns exist on the Rust `Template` struct but not in the query, causing a SQLx `ColumnNotFound` decoding error at runtime. The handler maps ALL errors to `AppError::NotFound`, so the decode error surfaces as a 404 instead of a 500.

Additionally, `get_template_by_id` has no fallback mechanism — when listing uses fallback templates (empty DB), the random UUIDs generated by `Template::fallback()` don't exist in the DB, so GET by UUID returns None → 404.

**Recommendation:** Fix the `get_template_by_id` query to select all struct fields, add a migration to add `version` column to the DB, improve error differentiation in the handler, and add fallback-by-id lookup in the repository.

## Root Cause Analysis

### Issue 1: Missing columns in `get_template_by_id` SELECT
```sql
-- Current query (missing `version` and `usage_count`):
SELECT id, game_type, category, display_name, description, config,
       visibility, user_id, is_builtin, is_active,
       created_at, updated_at
FROM templates WHERE id = $1
```

The `Template` struct expects 14 fields but the query only returns 12. The `version` column doesn't exist in the DB schema (no migration), and `usage_count` isn't selected. `usage_count` has `#[sqlx(default)]` but `version` is an `Option<String>` without `#[sqlx(default)]` — causing `ColumnNotFound` at runtime.

### Issue 2: Error masking
The handler maps all errors to 404:
```rust
.map_err(|_| AppError::NotFound)?;
```
This means a `ColumnNotFound` decode error (which is a 500-level bug) silently becomes a 404, making the true cause invisible.

### Issue 3: No fallback in `get_template_by_id`
`list_public_templates()` uses `Template::fallback()` when DB is empty, but `get_template_by_id()` has no fallback. If templates are displayed from fallback (random UUIDs), clicking Edit navigates to a UUID that doesn't exist in the DB.

## Fix Plan

### Fix 1: Add `version` column to templates table
Create a new migration: `ALTER TABLE templates ADD COLUMN IF NOT EXISTS version VARCHAR(50);`

### Fix 2: Update `get_template_by_id` SELECT
Add `version` to the SELECT clause. The `usage_count` field has `#[sqlx(default)]` so it can stay unselected (defaults to 0), but adding it explicitly is cleaner.

### Fix 3: Improve error handling
Differentiate between "not found" (None returned from repository) and "internal error" (SQL/decoding error). Map only None → 404, SQL errors → 500.

### Fix 4: Optionally add fallback for single-template GET
If listing uses fallback, the edit flow still fails. Consider adding fallback-by-game_type+category lookup or skipping fallback entirely when DB is populated.

## Common Pitfalls

### Pitfall 1: Silently swallowing errors
**What goes wrong:** `map_err(|_| AppError::NotFound)` hides the true error type
**How to avoid:** Match on the error kind — `NotFound` vs `InternalError`

### Pitfall 2: Struct-query mismatch
**What goes wrong:** Adding fields to the Rust struct without updating SQL queries
**How to avoid:** Use sqlx compile-time macros (`query_as!`) that catch mismatches at build time

## Code Examples

### Current (broken) handler pattern:
```rust
let template = use_case.execute(id).await
    .map_err(|_| AppError::NotFound)?;
```

### Fixed handler pattern:
```rust
let template = use_case.execute(id).await
    .map_err(|e| {
        if e.to_string().contains("not found") {
            AppError::NotFound
        } else {
            AppError::InternalError(e)
        }
    })?;
```

### Migration to add version column:
```sql
ALTER TABLE templates ADD COLUMN IF NOT EXISTS version VARCHAR(50);
```

### Updated get_template_by_id query:
```sql
SELECT id, game_type, category, display_name, description, config,
       visibility, user_id, is_builtin, is_active,
       created_at, updated_at, version
FROM templates WHERE id = $1
```

## Open Questions

1. **Does `version` column already exist in production DB?** — The original migration doesn't include it. If a manual ALTER was run, the column exists but the query still needs updating.
2. **Should we keep the fallback for `get_template_by_id`?** — If the DB is properly seeded, fallback isn't needed. But for dev environments without seed data, it would help.
3. **Is the delete flow also affected?** — `delete_template` also uses `get_template_by_id` internally (to check ownership before deleting), so the same column mismatch would break deletion too.
