<objective>
Adapt the gateway (`opt/relay/src/auth.rs`, `tunnel.rs`, `types.rs`) for Phase 69's per-server relay tunnel architecture:

1. **auth.rs 1:N mapping:** Change `authorize()` to return `Vec<ServerMapping>` instead of a single `ServerMapping` (or `Option<ServerMapping>`). One relay token can authorize multiple servers — each with its own `server_id` and `subdomain`.

2. **tunnel.rs N WS per agent IP:** Remove or relax the unique-agent-IP enforcement so the gateway accepts multiple WebSocket connections from a single agent IP (one per per-server relay client instance).

3. **types.rs:** Add `ServerMapping` struct if it doesn't already exist, used as the return type for `authorize()`.

Purpose: Phase 69 creates one yamux session per server from the same agent process. The gateway currently assumes one relay tunnel per agent (1:1 relay_token→server_id) and may enforce 1 WS connection per agent IP. Both assumptions must be relaxed for per-server tunnels.

Output:
- `auth.rs`: Vec<ServerMapping> return type, backward-compatible for single-server agents
- `types.rs`: ServerMapping struct (server_id: Uuid, subdomain: Option<String>)
- `tunnel.rs`: relaxed agent IP uniqueness check
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Agent → Gateway WS | 1:N mapping means one agent credential authorizes N tunnels |
| Gateway → Backend HTTP | authorize() call — response now returns Vec of server mappings |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-69-13 | S | auth.rs 1:N return | mitigate | Backend HTTP response shape changes from single object to JSON array. Gateway validates response as `Vec<ServerMapping>` with serde — unexpected shape causes error, not misinterpretation. |
| T-69-14 | T | auth.rs parameter removal | mitigate | Removing `server_id` from `authorize()` means the backend decides which servers the agent manages. If an agent was compromised, it could previously only tunnel for one server; now it can tunnel for all of the agent's servers. Mitigation: backend authorization already validates agent→server ownership per Phase 68. This is not a new trust surface. |
| T-69-15 | A | tunnel.rs N WS per IP | accept | Removing agent IP uniqueness could theoretically enable one agent to open unlimited connections, but each connection requires a valid relay token authorizing a distinct server_id. The registry's per-server_id uniqueness already prevents two tunnels for the same server. The agent already controls all its servers' tunnels. |
</threat_model>
