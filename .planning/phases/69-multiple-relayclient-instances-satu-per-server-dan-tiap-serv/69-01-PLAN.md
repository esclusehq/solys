<objective>
Add the `servers.subdomain` column to the database and wire subdomain generation into the server creation flow. When a server is created, the backend generates a short hex hash of the server UUID (e.g. `a3f8b`), stores it, and includes it in every `relay.connect` task payload. On agent WS reconnect, the backend re-pushes `relay.connect` for all active servers the agent owns. On server deletion, it pushes `relay.disconnect`.

Purpose: Phase 69 requires every server to have a unique subdomain so the gateway's Minecraft Handshake parser (already implemented in Phase 68 player.rs) can route player connections to the correct per-server tunnel. The subdomain must be deterministic (derived from the immutable server UUID) and generated at server creation time.

Output:
- One forward migration adding `servers.subdomain` column + unique index
- Server entity extended with `subdomain` field
- Repository trait + implementation for subdomain generation (blake3/SHA-256 first 3 bytes → 5 hex chars)
- Server creation handler generates and stores subdomain
- relay.connect task payload extended with `server_id, subdomain, public_port, local_mc_addr`
- Node WS registration handler: on Register, push relay.connect for all servers
- Server deletion handler: push relay.disconnect
</objective>

Subdomain generation algorithm (from RESEARCH.md):
```rust
// Deterministic short hex hash of server UUID.
// 3 bytes → 6 hex chars → 16M namespace.
// Collision probability at 100K servers: ~0.01%
fn generate_subdomain(server_uuid: &Uuid) -> String {
    use sha2::{Sha256, Digest};
    let hash = Sha256::digest(server_uuid.as_bytes());
    hex::encode(&hash[..3])  // 6 hex chars
}
```
