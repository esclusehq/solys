**Gathered:** 2026-06-09
**Status:** Ready for planning

<domain>
## Phase Boundary

Refactor the agent's relay tunnel architecture from a single global tunnel (one WebSocket + yamux session per agent, all servers multiplexed) to per-server tunnel instances. Each server on a node gets:

1. **Its own RelayClient** — independent WS connection to relay.esluce.net, independent yamux session, independent reconnect loop
2. **Its own unique subdomain** — short hex hash on `*.play.esluce.net` (e.g., `a3f8b.play.esluce.net`), generated by the backend on server create, stored in `servers.subdomain`
3. **Gateway Handshake-based routing** — wildcard `*.play.esluce.net` → NLB → gateway, which parses the Minecraft Handshake packet's server address field to extract the subdomain and route to the correct tunnel

**Out of scope:**
- Multi-region relay failover (deferred from Phase 68)
- Bedrock Edition UDP support (architecturally different)
- Backend auth layer for one relay_token → many server_ids (gateway change)
</domain>

<decisions>
## Implementation Decisions

### Per-Server Tunnel Instantiation
- **D-01 (Storage):** `tokio::sync::RwLock<HashMap<ServerId, PerServerRuntime>>` — concurrent reads (heartbeats, player streams) with exclusive writes (add/remove server). Fine at agent scale (tens of servers, not hundreds).
- **D-02 (Add/remove servers):** Via backend task dispatch — `relay.connect` and `relay.disconnect` tasks with `server_id` in payload. Agent adds/removes HashMap entries on arrival. No agent-side reconciliation.
- **D-03 (PerServerRuntime struct):** Full per-server struct containing: `CancellationToken` (child of parent), `JoinHandle`, `ControlTx` (mpsc channel), `BytesCounter` (AtomicU64), `TunnelStart` (Instant), and per-server `RelayConfig` (server_id, subdomain, token, gateway_url, local_mc_addr, public_port).
- **D-04 (Shutdown cascade):** Parent `CancellationToken` held at `RelayRuntime` level. Each `PerServerRuntime` gets a `child_token()`. On agent shutdown: `parent.cancel()` fires all children, all reconnect loops exit naturally, then HashMap is cleared. Tokio's CancellationToken ref-counting ensures correct behavior (explicit parent cancel, not relying on drop).

### Task Dispatch with server_id
- **D-05 (Task routing):** `server_id` carried in `task.payload` for `relay.connect`, `relay.disconnect`, `relay.heartbeat` tasks. Agent extracts and dispatches to correct `PerServerRuntime`.
- **D-06 (Duplicate tunnel):** Replace existing — cancel old tunnel atomically, start new one. Matches Phase 68 D-21 replacement semantics.
- **D-07 (Agent reconnect):** On agent → backend WS reconnect, backend pushes `relay.connect` for all active servers the agent owns. Backend is the source of truth; no local persistence.
- **D-08 (Server deletion):** Backend sends `relay.disconnect` when a server is deleted. Agent cancels tunnel, removes from HashMap, cleans up DNS records.

### Subdomain Generation
- **D-09 (Format):** Short hex hash of server UUID (e.g., `a3f8b`). Predictable, collision-resistant, short enough for chat sharing. Not server-name-derived (name changes would break subdomain stability).
- **D-10 (Generation timing):** Backend generates subdomain on server create. Stored in `servers.subdomain` column. Passed to agent in `relay.connect` task payload.
- **D-11 (DNS):** Wildcard `*.play.esluce.net` → CNAME/ALIAS → NLB. No per-record DNS management needed. The wildcard already resolves all subdomains to the gateway's NLB; the gateway uses the Handshake parser, not DNS records, to determine routing.
- **D-12 (Gateway routing):** Gateway parses the Minecraft Handshake packet's server address field to extract the full subdomain. Maps subdomain to `server_id`, looks up active tunnel in registry, opens a yamux stream in that tunnel's session. No DNS-per-record overhead.

### Per-Server Tunnel Config Source
- **D-13 (Config delivery):** Backend includes full per-server config (server_id, subdomain, token, local_mc_addr, public_port) in `relay.connect` task payload. No separate config message.
- **D-14 (Persistence):** No local persistence. On agent restart → backend WS reconnect → backend re-pushes `relay.connect` for all servers. No caching in `state.json`.
- **D-15 (Shared vs per-server fields):** Shared fields (gateway_url, agent_public_ip, region, dns_api_token) remain as global config from env vars (existing `RELAY_CONFIG` OnceCell). Per-server fields arrive in task payload. Agent merges on connect.

### Resource Isolation & Limits
- **D-16 (Max tunnels):** No hard limit per agent. At typical scale (~tens of servers per node), resource overhead is negligible (each tunnel ~few MB + one WebSocket + one yamux session + one heartbeat task).
- **D-17 (Heartbeat staggering):** Random startup delay (0-10s jitter) per tunnel so heartbeats are evenly distributed across the 10s interval. Avoids thundering herd at the gateway.
- **D-18 (Bandwidth accounting):** Per-server `AtomicU64` in each `PerServerRuntime`. Gateway receives per-server byte counts in tunnel heartbeats for per-server monitoring.

### Gateway Changes
- **D-19 (Auth layer):** `relay_token` (per-node) must authorize multiple `server_id` values (all servers owned by the node). Backend's `/internal/relay/authorize` endpoint validates `node.owns(server_id)` — no change needed at the endpoint, but the gateway's auth check in `auth.rs` must not assume 1:1 token→server_id.
- **D-20 (WS connection management):** Gateway must accept N concurrent WS connections from the same agent IP (one per server), each authenticating with the same `relay_token` but different `server_id`. Connection management in `tunnel.rs` must not conflict on agent identity.
- **D-21 (Registry):** No change — already `HashMap<ServerId, TunnelHandle>` per-server.
- **D-22 (Player routing):** No change — already routes by subdomain → server_id via Handshake parser.

### the agent's Discretion
- Exact short-hex length (recommend 4-6 hex chars from server UUID hash)
- Heartbeat jitter formula and exact startup delay distribution
- PerServerRuntime struct field ordering and naming
- HashMap eviction policy (if any)
- Whether to log per-server tunnel state transitions (suggest: yes, per existing audit pattern)
- Whether the gateway's auth layer handles per-server WS via relay_token dedup at the TCP level (recommend: accept N connections, each authenticates independently)
</decisions>

<specifics>
## Specific Ideas

### Architecture change from Phase 68
```
Phase 68 (single tunnel):
  Agent ──[1 WSS]──> Gateway ──[1 yamux session for ALL servers]──>
    Player A → stream for server_1
    Player B → stream for server_2
    Player C → stream for server_1

Phase 69 (per-server tunnels):
  Agent ──[WSS for server_1]──> Gateway ──[1 yamux session for server_1]──>
  Agent ──[WSS for server_2]──> Gateway ──[1 yamux session for server_2]──>
    Player A → yamux stream in server_1 session
    Player C → yamux stream in server_1 session
    Player B → yamux stream in server_2 session
```

### Gateway routing flow (unchanged from Phase 68)
```
Player connects to a3f8b.play.esluce.net:25565
  ↓
NLB forwards raw TCP to gateway:25565
  ↓
Gateway receives TCP, reads first bytes (Minecraft Handshake packet)
  ↓
Gateway parses Handshake → extracts "a3f8b.play.esluce.net" from server address field
  ↓
Gateway looks up server_id by subdomain mapping
  ↓
Gateway looks up active tunnel for server_id in registry
  ↓
Gateway opens yamux stream in that tunnel's session
  ↓
Bidirectional copy between player TCP and yamux stream
  ↓
Agent's per-server yamux session receives stream
  ↓
Agent dials localhost:25565 (or per-server port)
  ↓
Bidirectional copy between yamux stream and local TCP
```

### Shutdown cascade (parent-child CancellationToken)
```
RelayRuntime {
    shutdown: CancellationToken,    // parent
    tunnels: Arc<RwLock<HashMap<...>>>
}

async fn shutdown() {
    shutdown.cancel();              // all child tokens fire
    tunnels.write().await.clear();  // drops PerServerRuntime structs, JoinHandles detach
}

// Each PerServerRuntime:
PerServerRuntime {
    cancel: shutdown.child_token(), // child of parent
    join: JoinHandle,
    ...
}
```
</specifics>
