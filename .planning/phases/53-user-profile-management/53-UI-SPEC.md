## Focal Point

The primary visual anchor on the Profile tab is the **avatar upload area** (96×96px circular region at the top of the page). Its size, circular crop, and `--color-cosmic-cyan` glow on interaction draw the eye first. Visual hierarchy flows downward naturally:

1. Avatar upload area (primary anchor)
2. Display Name field (first interactive form element)
3. Existing Profile info (Name, Email — unchanged)
4. Existing Change Password section (unchanged)
5. Login History table (read-only information section)
6. Delete Account section (destructive action, bottom)

This order ensures the user encounters the most personal/visual element first, then the editable field, then informational history, and finally the rare destructive action.

## Component Inventory

Components defined for this phase. Each extends the existing `renderProfileTab` function in `SettingsPage.jsx`.

### 1. AvatarUpload
- **Type:** Inline section within profile tab
- **Purpose:** Upload and display user avatar
- **Interaction:**
  - Click avatar area → OS file picker → validate → auto-upload to Supabase Storage
  - Drag image onto avatar area → validate → auto-upload
  - No crop step (D-02)
- **File validation (client-side):** JPG/PNG/WebP, max 2MB (D-03)
- **Visual:**
  - Circular 96×96px area with current avatar or placeholder icon
  - Hover: subtle `--color-cosmic-cyan` border glow, overlay text "Change avatar"
  - Uploading: spinner overlay or skeleton pulse
  - Error state: red border + toast error message
  - Success: green brief flash + toast
- **States:**
  - **Empty (no avatar):** Circular dashed border area with upload icon + "Click or drag image here" text
  - **Has avatar:** Display current avatar image, overlay "Change avatar" on hover
  - **Uploading:** Spinner/loading indicator on avatar area
  - **Error (invalid file/size):** Toast error, highlight border red briefly
  - **Success:** Avatar image updates immediately, success toast "Avatar updated"

### 2. DisplayNameField
- **Type:** Form field within profile tab
- **Purpose:** Edit display name (independent from login identifier)
- **Interaction:** Text input → save button → PUT /api/profile or Supabase user metadata update
- **Visual:** Standard form pattern:
  - Label: "Display Name"
  - Helper text: "Shown in the sidebar" (`text-xs text-gray-500`)
  - Text input: `bg-gray-700 text-white rounded focus:ring-2 focus:ring-blue-500`
  - Save button: `px-4 py-2 bg-blue-600 text-white rounded hover:bg-blue-700`
- **States:**
  - **Default:** Pre-filled with current display name (or empty for first-time)
  - **Saving:** Button shows "Saving...", disabled
  - **Error:** Toast error, field retains value
  - **Success:** Toast "Display name updated", button resets

### 3. LoginHistoryTable
- **Type:** Table section within profile tab
- **Purpose:** View recent login activity (90-day retention)
- **Columns:** Timestamp, IP address, Device/Browser, OAuth Provider, Session ID
- **Interaction:** Scrollable table (read-only)
- **Visual:**
  - Section header "Login History" (`text-lg font-semibold text-white`)
  - Table with headers in `text-gray-400`, rows in `text-gray-300`
  - Row hover: `bg-[rgba(255,255,255,0.03)]`
  - No built-in pagination (show all within 90-day window; if >50 rows, can add later)
- **States:**
  - **Loading:** Skeleton loader rows (3-4 gray pulsing lines)
  - **Empty:** "No login history found" centered text with muted icon
  - **Populated:** Table with data rows
  - **Error:** "Could not load login history. Please try again." + retry button

### 4. DeleteAccountSection
- **Type:** Destructive action section at bottom of profile tab
- **Purpose:** Initiate account deletion with safety confirmation
- **Interaction:** Two-step flow:
  1. Click red "Delete Account" button → expand confirmation panel
  2. Re-enter password → type "DELETE" → enable final red "Permanently Delete" button
- **Visual:**
  - Section separated by `border-t border-gray-700 pt-6` with red-tinted left border accent
  - Warning text in red/orange tones
  - Re-auth input field
  - Type-confirmation input field
  - Submit button: `bg-red-600 text-white` → disabled until both conditions met
- **States:**
  - **Default:** Collapsed section with:
    - "Danger Zone" heading in red text
    - Warning paragraph about consequences
    - Red "Delete Account" button
  - **Confirming (expanded):**
    - "Enter your password to continue" input
    - "Type DELETE to confirm" input
    - "Permanently Delete Account" button (disabled until password valid + typed "DELETE")
  - **Submitting:** Button shows "Processing...", all inputs disabled
  - **Error:** Toast with error message
  - **Success:** Redirect to logout/signed-out page (account soft-deleted)

### 5. TransferOwnershipLink
- **Type:** Within delete account section
- **Purpose:** Allow server transfer before deletion deadline
- **Interaction:** Link/button in delete section → navigates to team management or opens a transfer flow
- **Visual:** Text link "Transfer ownership of your servers" with icon
- **States:**
  - **Has servers:** Show link (active)
  - **No servers:** Hide or show muted text "No servers to transfer"
