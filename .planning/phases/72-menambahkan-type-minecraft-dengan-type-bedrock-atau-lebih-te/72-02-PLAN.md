<objective>
Enable the Esluce agent to create and start Docker containers with UDP port bindings when the server type is Bedrock, and use the correct port map key.

**Purpose:** Minecraft Bedrock Edition uses UDP protocol (port 19132) instead of TCP (port 25565). The agent currently hardcodes TCP for all port bindings and hardcodes `"25565"` as the port map key. These two changes make the agent protocol-aware so Bedrock containers work correctly.

**Output:** Modified agent_connection.rs (forward loader + dynamic port key) and runtime.rs (UDP dispatch in create + start handlers).
</objective>

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-72-05 | Tampering | `agent_connection.rs` loader field injection | mitigate | loader value comes from API's DeployConfig which is set by `agent_server_executor.rs` based on `server.mc_loader`. The `mc_loader` is set by the use case which validates against known values. If an attacker sends a crafted WebSocket message with arbitrary loader, the worst outcome is wrong protocol (UDP vs TCP) — container still starts, just wrong port binding. |
| T-72-06 | Spoofing | `runtime.rs` port binding protocol | mitigate | Protocol selection is derived from loader field, which is server-controlled. An attacker spoofing loader="bedrock" for a Java server would cause UDP binding instead of TCP — Java clients would fail to connect. Mitigation: loader is set server-side, not from user input. The frontend game_type selection is validated by the use case. |
| T-72-07 | Denial of Service | Malformed port map key | mitigate | Port key is `format!("{}/udp", container_port)` which is a valid Docker port spec. Bollard API accepts `/udp` and `/tcp` suffixes. Invalid ports (<1, >65535) are rejected by Docker daemon before container creation. |
</threat_model>
