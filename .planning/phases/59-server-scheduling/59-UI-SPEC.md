## Interaction Design

### Add/Edit Schedule Flow

1. User clicks `+ Add Schedule` (or Edit on existing row)
2. Inline form expands below section header (schedule list shifts down)
3. Form contains:
   - **Action Type** dropdown: Start, Stop, Restart, Sleep
   - **Schedule** dropdown + text input:
     - Preset options: "Every hour", "Every 6 hours", "Every 12 hours", "Daily at midnight", "Weekly on Sunday", "Custom..."
     - When "Custom..." selected, cron text input appears
   - **Timezone** dropdown: Common timezones (UTC, America/New_York, America/Los_Angeles, Europe/London, Europe/Berlin, Asia/Tokyo, Asia/Jakarta, Asia/Singapore, Asia/Shanghai, Australia/Sydney) + text input for any IANA zone
   - **Run Once** checkbox with label "Run once and disable after execution"
4. User clicks "Save" ‚Äî validates on client:
   - Action type required
   - Cron expression valid (non-empty, standard 5-field format)
   - Timezone valid (IANA timezone or empty = UTC)
5. On save: API call ‚Üí success toast ‚Üí form closes ‚Üí list refreshes
6. On cancel: form closes, unsaved changes discarded (no confirmation ‚Äî form is inline, low stakes)

### Toggle Enabled/Disabled

- Click ON/OFF label button ‚Üí immediate disable/enable
- No confirmation dialog (low-stakes toggle)
- Calls `PATCH /tasks/:id { enabled: bool }`
- Optimistic UI update with rollback on error

### Delete Schedule Flow

1. User clicks red "Del" button on schedule row
2. Confirmation modal appears (matches existing pattern from ScheduledTasksPage.jsx)
3. "Delete" button in modal ‚Üí calls `DELETE /tasks/:id` ‚Üí removes row ‚Üí toast
4. "Cancel" button ‚Üí modal closes

### Last-Result Display

- Schedule row shows last-run info on second line: `Last: {date} ‚úì` / `Last: {date} ‚úó` / `Last: {date} ‚è≠` / `Never run`
- On failure (`last_error` populated): row shows error message below the schedule info in red text
- Error text: truncated to single line, full message visible on hover/tooltip

## Key Design Decisions

| Decision | Value | Rationale |
|----------|-------|-----------|
| Inline form vs modal | **Inline expandable form** | Matches the pattern of other settings sections; avoids modal stacking; form is simple (5 fields) |
| Schedule format display | Human-readable text (e.g., "Daily at 08:00") + raw cron on hover | Provides instant clarity for common schedules, detail on demand |
| Timezone input | Dropdown with 10 common zones + free-text IANA input | Common zones cover 95% of use cases; free-text handles edge cases |
| Action type badge colors | Semantic: green=start, red=stop, amber=restart, purple=sleep | Immediate visual recognition of action type |
| Toast integration | Use existing `useUIStore.addToast()` | Consistent with existing error/success notification pattern |
| Delete confirmation | Modal dialog | Matches existing `ScheduledTasksPage.jsx` delete pattern; prevents accidental deletion |
| Optimistic toggling | Immediate UI update on enabled toggle | Low risk (toggle is easily reversed); better UX than waiting for API |
