# 11-01-PLAN.md — Fix API Routes Versioning

## Summary

Fix all API routes to use consistent `/api/v1/` versioning prefix, mirroring the structure already in place in `api_routes.rs` via nested handlers.

## Problem

Currently, `server_routes.rs` and `node_routes.rs` define routes without the `/api/v1/` prefix (e.g., `/api/servers/:id/logs`), while `api_routes.rs` uses `.nest("/api/v1/servers", ServerHandlers::router(...))` which already provides proper versioning.

This creates inconsistency and potential duplicate routes.

## Routes Without Versioning (to be fixed)

### server_routes.rs (~40 routes)
- `/api/test`
- `/api/servers/...` (many endpoints)
- `/api/plugins/...`
- `/api/alert-rules/...`
- `/api/alert-history`
- `/api/settings/...`
- `/api/deploy/...`
- `/api/runtimes`

### node_routes.rs (~15 routes)
- `/api/ws/node`
- `/api/nodes/...`

## Solution

1. **Identify duplicates**: Check if routes in `server_routes.rs` and `node_routes.rs` already exist in their respective Handler routers
2. **Remove duplicates**: Delete routes that are already covered by `ServerHandlers::router()`, `NodeHandlers::router()`, etc.
3. **Add remaining to handlers**: For routes that don't exist in handlers, add them there
4. **Remove unused route files**: Once migrated, remove or disable `server_routes.rs` and `node_routes.rs` from bootstrap

## Execution

### Step 1: Audit routes in handlers

Check what routes exist in each Handler's `router()` function vs what's in `server_routes.rs` and `node_routes.rs`:

- `ServerHandlers::router()` in `server_handlers.rs`
- `NodeHandlers::router()` in `node_handlers.rs`

### Step 2: Remove duplicate routes

Routes that exist in both places should be removed from `server_routes.rs` and `node_routes.rs`.

### Step 3: Add missing routes to handlers

Routes that don't exist in handlers but are needed should be added to the appropriate Handler router.

### Step 4: Update bootstrap

Remove `.merge(server_routes::routes(...))` and `.merge(node_routes::routes(...))` from `api/src/bootstrap/mod.rs`.

### Step 5: Verify

- Ensure all endpoints work with `/api/v1/...` prefix
- No 404s on existing endpoints
- API documentation reflects correct paths
