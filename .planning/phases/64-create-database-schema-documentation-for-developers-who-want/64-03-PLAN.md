  <name>Task 2: Run generator to produce DATABASE_SCHEMA.md</name>

  <read_first>
    - ARCHITECTURE.md (reference: repo-root doc format — badges, TOC style, ATX headings, GFM tables)

  <files>
    DATABASE_SCHEMA.md
  </files>

  <action>
    1. Verify PostgreSQL is accessible (the tool will use DATABASE_URL from environment):
    ```bash
    # Check that DATABASE_URL is set or can be derived
    if [ -z "${DATABASE_URL}" ]; then
      # Try to find DATABASE_URL from docker compose or .env files
      if [ -f api/.env ]; then
        export DATABASE_URL=$(grep -oP 'DATABASE_URL=\K.*' api/.env | head -1)
      fi
    fi
    echo "DATABASE_URL: ${DATABASE_URL:-NOT SET}"
    ```

    2. If `supabase start` is running, try to get the DB URL:
    ```bash
    # Try supabase status to get db url
    supabase status 2>/dev/null | grep 'DB URL' | awk '{print $NF}'
    ```

    3. Run the generator:
    ```bash
    cargo run --manifest-path tools/db-schema-gen/Cargo.toml -- \
      --db-url "${DATABASE_URL}" \
      --source-dir api/src/domain \
      --output DATABASE_SCHEMA.md
    ```

    4. If the generator fails at runtime, fix issues:
       - Connection errors: verify PostgreSQL is running (`docker compose ps postgres`)
       - Schema introspection errors: verify tables exist (`psql "$DATABASE_URL" -c "SELECT table_name FROM information_schema.tables WHERE table_schema='public'"`)
       - If table names mismatch domain classification, update `domain_classifier.rs`
       - If the generator crashes with a panic, check for unwrap() calls on Options and fix with proper error handling

    5. Verify the generated output:
    ```bash
    # File exists and is non-trivial
    wc -l DATABASE_SCHEMA.md
    test -s DATABASE_SCHEMA.md
    
    # Contains required sections
    grep -c '# Escluse — Database Schema' DATABASE_SCHEMA.md
    grep -c '```mermaid' DATABASE_SCHEMA.md
    grep -c 'erDiagram' DATABASE_SCHEMA.md
    grep -c '| Column | Type | Constraints | Description |' DATABASE_SCHEMA.md
    
    # Contains table content (not just headers)
    grep -c "^\`" DATABASE_SCHEMA.md
    
    # Contains Migration History section
    grep -c 'Migration History' DATABASE_SCHEMA.md
    ```

    6. If the generated output is valid (contains content and required sections), verify the Mermaid diagrams will render on GitHub:
       - No empty entity blocks `{}`
       - All `erDiagram` blocks have at least one entity definition before any relationship line
       - Relationship syntax matches: `{entity} {cardinality} {entity} {cardinality} : {label}`
       - No unclosed backtick fences

    7. If DATABASE_SCHEMA.md was already generated by a previous run, compare and ensure the new version is not empty or truncated (should be 500+ lines).

    **If no PostgreSQL is available** (development environment not running), create an initial `DATABASE_SCHEMA.md` manually that documents all tables based on migrations + entity structs, following the exact same format the generator would produce. Mark the note header as:
    ```

    ```
    And add a warning callout:
    ```
    > **Note:** This initial version was created from migration files and entity structures.
    > Regenerate from a live database for the most accurate snapshot:
    > ```bash
    > cargo run --manifest-path tools/db-schema-gen/Cargo.toml -- --db-url "$DATABASE_URL"
    > ```
    ```
    The manual version should include:
    - All tables from migrations (69 migration files)
    - All column names, types, constraints from migration SQL
    - Mermaid ER diagrams for each relationship cluster (10 clusters from RESEARCH.md)
    - GFM tables per table
    - Domain descriptions from research
    - Query patterns and design rationale from entity annotations  
    - Migration History cross-reference
  </action>
