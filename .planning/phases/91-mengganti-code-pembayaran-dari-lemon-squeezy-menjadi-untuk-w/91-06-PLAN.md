In `app/src/pages/billing/BillingPage.jsx`, find the section that renders plan cards (likely a `.map()` over `plans` data) and make the following changes per UI-SPEC:

### 1. Remove "Locked" and "Coming Soon" badges
Find the badge elements (likely `<div>` or `<span>` with classes like `bg-gray-500/20 text-gray-400` for "Locked" and `bg-[var(--color-cosmic-orange)]/20 text-[var(--color-cosmic-orange)]` or similar for "Coming Soon"). Remove these badge elements entirely from the plan card rendering.

### 2. Enable the CTA buttons
Find the button logic that currently renders disabled buttons with "Locked" or "Coming Soon" text. Replace with the UI-SPEC button logic:

- **No active subscription** (user is on Free plan or subscription is canceled/expired): Show **"Subscribe"** button for each plan card
- **User has an active subscription** (status is active or trialing):
  - For the plan the user is currently subscribed to: Show "Current Plan" label (may already exist — keep as-is)
  - For higher-tier plans: Show **"Upgrade to {plan.display_name}"** button
  - For lower-tier plans: Show **"Downgrade to {plan.display_name}"** button (disabled if past_due per D-23)
- **User subscription status is past_due**: All plan upgrade buttons are disabled (`opacity-40 cursor-not-allowed`) with tooltip showing "Update payment to change plan"

### 3. Button click handler for Subscribe/Upgrade
When the user clicks "Subscribe" or "Upgrade to X", call the existing `handleUpgrade` function (or create it if it doesn't exist). The function should:
1. Set a loading state (replace button text with spinner or set `opacity-50` with text "Mengarahkan...")
2. Call `billingApi.createCheckout(planId, billingCycle)` (API already exists — uses the refactored backend from Plan 03)
3. On success: `window.location.href = checkoutUrl` (redirects to Whop checkout page)
4. On error: show error toast via `useUIStore.addToast`

### 4. Button styling (per UI-SPEC Interaction States)
- Default: `bg-[var(--color-cosmic-cyan)] text-white hover:opacity-80 transition-opacity duration-150`
- Loading: `opacity-50 cursor-wait` with text "Mengarahkan..." OR inline spinner component
- Disabled (past_due): `opacity-40 cursor-not-allowed`
- Error: Toast notification with `type: 'error'`

### 5. Check for existing `handleUpgrade` function
If `handleUpgrade` already exists in the component (e.g., from previous LS integration), it likely takes a `planId` parameter. Verify it calls `billingApi.createCheckout(planId, billingCycle)` and not an LS-specific endpoint. If it still references LS, update it to use the generic `billingApi.createCheckout` call.

### 6. Default billing cycle
If the user is on a monthly subscription, offer monthly upgrade. If yearly, offer yearly. Default to monthly if no subscription exists. The UI already likely has a monthly/yearly toggle — respect whatever the user has selected.

### Location
Insert the banner at the top of the billing page content area, above the plan cards grid. It should only render when `subscription.status === 'past_due'`.

### Visual design (per UI-SPEC §Copywriting Contract and §Interaction States)
```jsx
{subscription?.status === 'past_due' && (
  <div className="mb-6 p-4 rounded-lg border border-[var(--color-cosmic-orange)]/30 bg-[var(--color-cosmic-orange)]/5 backdrop-blur-sm">
    <div className="flex items-start justify-between">
      <div className="flex-1">
        <h3 className="text-sm font-semibold text-[var(--color-cosmic-orange)] mb-1">
          Payment Failed
        </h3>
        <p className="text-sm text-gray-300 mb-3">
          Your subscription payment failed. You still have full access until{' '}
          {format(graceEndDate, 'MMMM d, yyyy')}. Please update your payment 
          method to avoid service interruption.
        </p>
        <button
          onClick={() => {
            if (subscription?.manage_url) {
              window.open(subscription.manage_url, '_blank');
            } else {
              useUIStore.getState().addToast({
                type: 'error',
                message: 'Customer portal not available. Please contact support.',
              });
            }
          }}
          className="px-4 py-2 text-sm font-medium rounded-lg border border-[var(--color-cosmic-orange)]/50 text-[var(--color-cosmic-orange)] hover:bg-[var(--color-cosmic-orange)]/10 transition-colors duration-150"
        >
          Update Payment Method
        </button>
      </div>
      <button
        onClick={() => setDismissBanner(true)}
        className="text-gray-500 hover:text-gray-300 transition-colors ml-4"
        aria-label="Dismiss"
      >
        <svg className="w-5 h-5" fill="none" viewBox="0 0 24 24" stroke="currentColor">
          <path strokeLinecap="round" strokeLinejoin="round" strokeWidth={2} d="M6 18L18 6M6 6l12 12" />
        </svg>
      </button>
    </div>
  </div>
)}
```

### Grace period end date
The grace period end date (`graceEndDate`) should be calculated from `subscription.current_period_end`:
```jsx
const graceEndDate = subscription?.current_period_end 
  ? new Date(subscription.current_period_end) 
  : new Date(Date.now() + 14 * 24 * 60 * 60 * 1000);
```
Per D-23, the grace period is 14 days from when payment failed. The `current_period_end` from the subscription approximates this. Use the subscription's current_period_end if available, otherwise add 14 days from now.

### Dismiss behavior
Add a state variable `dismissBanner` (default `false`). When user clicks the X button, set it to `true` and hide the banner. The banner reappears on page reload (since dismiss is local state).

### Plan buttons disabled during past_due
Ensure that when `subscription.status === 'past_due'`, all plan CTA buttons (from Task 1) have `disabled={true}` and `className` includes `opacity-40 cursor-not-allowed`. Add a tooltip/title attribute: `title="Update payment to change plan"`.

### Import check
If `format` from `date-fns` is used for date formatting, ensure it's imported or use a simple alternative like:
```jsx
const formattedDate = graceEndDate.toLocaleDateString('en-US', {
  month: 'long',
  day: 'numeric',
  year: 'numeric',
});
```
Prefer the `toLocaleDateString` approach to avoid adding a new dependency.
